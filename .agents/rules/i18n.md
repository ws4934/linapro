# `i18n` 治理规则

## 治理范围与影响评估

- 所有功能改动都必须评估对 `i18n` 的影响面，包括新增功能、修改现有功能、删除功能、调整菜单、路由、按钮、表单、表格、提示文案、接口文档、插件清单与初始化资源等场景。
- `i18n` 治理约束必须先区分宿主与插件，禁止把宿主的多语言要求无条件套到所有插件上。
- 若本次功能改动确认不影响 `i18n` 资源，也必须在任务执行或审查结论中明确记录该判断。
- 如果变更位于未启用 `i18n` 的插件且无需多语言资源，审查结果必须明确说明该结论。
- 提案、设计、实施、反馈、审查和归档前处理任何功能或治理变更时，都必须至少记录 `i18n` 影响判断；若判断存在影响，必须先读取并执行本文件中的宿主与插件边界、资源维护和验证要求。该要求由项目规范和项目自维护技能承载，不要求手动修改由 OpenSpec 工具维护的技能或 prompt 文件。

## 宿主与插件边界

- 宿主能力始终按宿主配置中的 `i18n.locales` 判断语言与资源治理范围，并同步维护前端运行时语言包、宿主 `manifest/i18n` 资源以及宿主 `apidoc i18n JSON`。
- 插件的多语言治理必须先读取插件的 `plugin.yaml`。只有显式配置 `i18n.enabled: true` 时，才按该插件的 `i18n.locales` 启用插件自身的多语言治理，并检查或补齐该插件自己的 `manifest/i18n` 与 `apidoc i18n JSON` 资源。
- `plugin.yaml` 缺少 `i18n`、缺少 `enabled` 或 `i18n.enabled` 不是 `true` 时，该插件视为单语言插件，框架、测试和审查必须跳过该插件自身的 `manifest/i18n` 与 `apidoc i18n JSON` 资源约束，不要求补翻译键、空占位文件或多语言目录。
- 在启用 `i18n` 多语言特性时，必须使用英文作为源内容语言；否则按用户指定语言或当前需求上下文语言生成单语言内容。

## 语言注册与资源布局

- 新增内置语言必须通过新增对应 `manifest/i18n/<locale>/*.json`、`manifest/i18n/<locale>/apidoc/**/*.json` 资源以及默认配置文件中的 `i18n.locales` 元数据完成。
- 禁止为了注册语言而修改后端 `Go` 枚举、宿主 `SQL seed` 或前端 `TypeScript` 语言清单。
- 宿主 `i18n.enabled=false` 时，前端必须隐藏语言切换并按 `i18n.default` 展示。
- 运行时翻译包缓存失效必须传入显式 `scope`，按语言、扇区、插件或业务内容范围精细失效，禁止在普通业务路径中无理由清空所有语言和所有扇区。

## API 文档本地化

- 宿主 API 文档按宿主 `i18n` 配置治理；插件 API 文档必须先读取插件自己的 `plugin.yaml`。
- 宿主启用 `i18n` 时，手写 `g.Meta` 的 `summary`、`description`、`dc` 等文档元数据，以及 DTO 字段的 `dc` 等说明文本，必须写入清晰、可读、可检索的英文源文本，并按宿主 `i18n.locales` 维护非英文 `apidoc` 翻译资源。
- 只有插件显式配置 `i18n.enabled: true` 时，`g.Meta` 的 `summary`、`description`、`dc` 等文档元数据，以及 DTO 字段的 `dc` 等说明文本，才必须写入清晰、可读、可检索的英文源文本并参与插件 `apidoc` 翻译治理。
- 未配置 `i18n`、缺少 `enabled` 或 `i18n.enabled` 不是 `true` 的插件，可以按自身源内容语言编写源文本，不要求补插件 `apidoc i18n JSON`。
- 任何场景都禁止写入 `api.user.list.title`、`{{t(...)}}` 这类不透明 `i18n` 占位符作为 OpenAPI 源文本。
- 该规则约束手写 API DTO 与固定路由投影文案；由数据表注释、GoFrame 或代码生成器产生的 schema 元数据进入接口文档时，`apidoc` service 层不得临时转换其语言，必须按数据源原文展示，也不得为 `internal.model.entity.*` 等生成 schema 键维护 `apidoc` 翻译项。

## API 文档资源隔离

- `OpenAPI/Swagger`、API 文档站点或接口调试页的本地化必须使用专门的 `apidoc i18n JSON` 资源维护，禁止复用或混入运行时前端 UI 的 `i18n` 语言包。
- 宿主接口翻译由 `apps/lina-core/manifest/i18n/<locale>/apidoc/**/*.json` 维护。
- 显式启用 `i18n.enabled: true` 的插件接口翻译必须由插件自己的 `apps/lina-plugins/<plugin-id>/manifest/i18n/<locale>/apidoc/**/*.json` 独立维护，禁止把 `plugins.*` 翻译键集中写入 `lina-core` 的 `apidoc` 资源。
- `lina-core` 的 `apidoc` 模块负责在渲染接口文档时按宿主、已启用 `i18n` 的插件、动态插件产物的顺序发现并合并这些资源。
- 参与 `i18n` 资源治理时，英文接口文档直接使用 API DTO 中的英文源文本，各级 `manifest/i18n/en-US/apidoc/**/*.json` 只保留空对象占位，不允许重新写入英文映射；未启用 `i18n` 的插件不要求提供这些占位文件。
- 若 GoFrame、生成的 `entity` schema 或数据表注释元数据进入接口文档，`apidoc` service 层不得维护中文到英文的临时转换表，也不得通过 `en-US/apidoc` 占位文件或非英文 `apidoc JSON` 翻译项兜底；需要改变其展示语言时，应修改对应数据源或生成源。
- 参与 `i18n` 资源治理且非英文文档源文本变更时，必须同步更新对应 `apidoc` 翻译资源。
- `eg` 示例值只作为真实请求或响应示例保留原值，不写入、不翻译 `apidoc i18n` 资源。

## API 文档翻译完整性

- 宿主或显式配置 `i18n.enabled: true` 的插件在新增或修改 API DTO 文档元数据时，必须补充或更新自动化测试、生成校验或审查脚本，确保英文源文本变化后不会遗漏任一非英文目标语言的 `apidoc` 翻译。
- 缺失翻译应被测试或审查流程阻断。
- 未启用 `i18n` 的插件不受该翻译完整性约束。
- `en-US` 空占位文件和 `eg/example` 示例值不纳入翻译覆盖要求。

## 运行时 UI 与字典文案

- 新增或修改菜单、路由、按钮、表单、表格、提示文案等用户可见内容时，必须评估 `i18n` 影响，并按宿主或插件的 `i18n` 启用状态同步维护对应资源。
- 未启用 `i18n` 的插件必须跳过插件自身的 `manifest/i18n` 与 `apidoc i18n JSON` 资源约束，不要求补翻译键、空占位文件或多语言目录。
- 业务模块的枚举值应通过字典模块维护字典类型和字典数据，而不是在代码中硬编码枚举值。

## 错误本地化

- 所有可能进入 HTTP API、插件调用、插件后端接口、WASM host service 或其他调用端响应载荷的业务错误、鉴权错误、参数错误和用户可见失败原因，都必须通过 `apps/lina-core/pkg/bizerr` 的 `NewCode`、`WrapCode` 或等价封装创建或包装。
- 调用端可见错误必须具备 GoFrame 类型错误码、稳定 `errorCode`、`messageKey`、`messageParams` 和英文 fallback。
- 各业务模块必须在所属模块的 `*_code.go` 中集中定义 `bizerr.Code`。
- 业务调用点禁止硬编码机器错误码、翻译键或裸错误文案。
- 调用端可见错误不得只依赖自由文本或 `gcode` 文案。

## 运行时依赖与服务边界

- `i18n` 属于关键运行期服务。宿主与插件的 `Controller`、`Middleware`、`Service`、插件宿主服务适配器和 `WASM host service` 必须通过构造函数参数逐项显式接收相关运行期依赖。
- 禁止在业务构造函数、请求处理路径、插件回调路径或 host service 调用路径中临时调用关键服务的 `New()` 创建独立服务图。
- 禁止通过 `Dependencies`、`Deps`、`Options` 等聚合结构体把多个接口对象或服务对象字段整体传递给依赖方；接口型依赖必须在构造函数签名中拆分为独立参数。
- 服务接口方法涉及权限、数据权限、租户隔离、缓存、`i18n`、事务、幂等、并发或外部资源访问时，接口注释必须说明调用方需要理解的边界和失败处理语义。
