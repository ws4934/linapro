# 接口契约规则

## 适用范围

本规则约束 HTTP API、RESTful 语义、路由、请求响应 DTO、OpenAPI/Swagger 元数据、接口权限标签、接口响应时间字段、接口性能契约和接口层文件组织。


## RESTful 设计要求

所有前后端 API 必须严格遵循 RESTful 设计规范，HTTP 方法与操作语义必须一一对应。

| HTTP 方法 | 语义 | 适用场景 |
|-----------|------|----------|
| `GET` | 读取，无副作用 | 列表查询、详情获取、树形数据、导出、下拉选项等所有只读操作 |
| `POST` | 创建资源或执行动作 | 新增记录、文件上传、导入、登录、登出等 |
| `PUT` | 更新资源 | 修改记录、状态变更、重置密码等 |
| `DELETE` | 删除资源 | 单条或批量删除 |

强制规则：

- 查询操作禁止使用`POST`。所有查询、列表、搜索、导出、获取详情等读操作必须使用`GET`，查询参数通过 URL query string 传递。
- 创建操作禁止使用`GET`。任何会产生副作用的操作必须使用`POST`。
- 删除操作必须使用`DELETE`，不允许用`POST`或`GET`执行删除。
- 更新操作使用`PUT`，不允许用`POST`修改已有资源。
- URL 设计使用名词复数或资源名，例如`/user`、`/dept`、`/dict/type`，避免使用动词，例如`/getUser`、`/deleteUser`。
- 子资源使用嵌套路径，例如`/dept/{id}/users`、`/user/{id}/status`。

## API 性能契约要求

- API 设计必须评估单次用户操作触发的 HTTP 调用次数和后端数据装配成本，不得通过接口契约迫使前端对列表每一行、树形每一节点或批量每一项再发起详情查询。
- 列表、导出、树形数据、聚合统计、下拉候选、批量获取等接口必须具备明确的分页、数量上限、范围过滤或异步处理边界，禁止无上限返回和无边界数据装配。
- 当页面需要展示关联对象名称、字典文案、权限状态、插件状态或统计值时，接口应提供最小必要投影或批量获取能力，避免仅返回 ID 后由前端逐项补查。
- 可选关联数据应通过明确的`expand`、`include`或独立批量接口表达，并要求后端使用批量装配；禁止把可选字段设计成隐含的逐项数据库查询。
- 接收多个 ID、资源键或节点键的查询参数必须有数量上限或分页机制，并在接口文档中说明超限行为。
- 懒加载树或分段加载可以用于控制单次数据规模，但必须保证每次加载有稳定边界，不得形成不可控的前端瀑布式调用链。

## API 响应时间字段契约

为满足多时区和前端页面差异化展示要求，公开 HTTP JSON 响应 DTO 中表示具体时间点的字段必须统一返回 Unix 毫秒时间戳，由前端根据页面和用户环境自行格式化展示。

强制规则：

- `createdAt`、`updatedAt`、`deletedAt`、`loginAt`、`startedAt`、`endedAt`、`expiredAt`、`lastRunAt`等表示具体时间点的响应字段，JSON 类型必须为数字，Go DTO 类型必须为`int64`或`*int64`。
- 公开响应 DTO 不得直接使用`time.Time`、`*time.Time`、`gtime.Time`、`*gtime.Time`作为时间点字段类型，也不得将时间点格式化为字符串返回。
- 规则仅约束 API 响应边界。`dao`、`do`、`entity`、`service`等内部模型仍可使用 Go 时间类型；在投影到公开响应 DTO 时完成 Unix 毫秒时间戳转换。
- `birthday`、`businessDate`、`periodDate`等只表示日历日期、不表示具体时间点的字段可以使用`YYYY-MM-DD`字符串，但字段`dc`或接口文档必须明确说明`date-only`语义，禁止让调用方推断服务端时区中的具体时间点。
- 时间点响应字段的`dc`或接口文档必须包含`Unix timestamp in milliseconds`说明，避免调用方混淆秒、毫秒或其他单位。

## 接口文件拆解要求

- 功能模型中，不要将该功能模块的所有接口都定义到一个 Go 文件中。
- 应按照接口用途拆分为不同 Go 文件，例如用户管理模块中的列表查询、详情、创建等接口应拆解到不同文件中。
- 接口文件拆解必须服务可读性和维护性，不得通过无意义拆分制造碎片化。

## 接口文档标签要求

所有 API 定义的结构体必须包含完善的文档标签，确保自动生成的 OpenAPI/Swagger 文档清晰、准确、可用。

- 输入 DTO 字段统一使用`json:"fieldName"`声明参数名，包括路径参数、Query 参数和请求体字段；禁止在输入结构体中混用`p`与`json`标签。
- 输出 DTO 字段继续使用`json:"fieldName"`标签，保持响应序列化和文档一致性。
- `g.Meta`必须包含`dc`标签，描述接口完整功能和使用场景，不得简单重复`summary`。
- 需要宿主统一权限校验的受保护静态 API 必须在`g.Meta`上声明`permission`标签。权限值使用与菜单/按钮权限标识一致的字符串，例如`permission:"plugin:install"`，由统一权限中间件执行校验；禁止在控制器方法内部重复手写同等语义的权限判断逻辑。
- 所有输入输出字段必须包含`dc`标签，对字段含义、取值说明、业务规则和关联关系进行清晰描述。
- 所有输入输出字段必须包含`eg`标签，提供真实可用的示例值。
- 状态、类型等枚举字段必须在`dc`中列出所有可选值及含义。
- 筛选条件等可选字段应说明不传时的默认行为。
- 涉及 API 文档本地化时，必须继续读取`.agents/rules/i18n.md`。

## 代码生成要求

- API 变更通常应先修改`api/{resource}/v1/*.go`，再按仓库生成流程执行`make ctrl`或等价入口。
- 不得手动修改由脚手架工具维护的接口方法定义生成文件，除非该文件本身就是受控的 API 源定义。
- 代码生成涉及后端 Go 或数据库时，还必须遵守`.agents/rules/backend-go.md`和`.agents/rules/database.md`。

## 验证要求

- API 变更必须运行覆盖变更包的 Go 编译门禁。
- 涉及 controller 构造、路由绑定或 API 接口签名时，必须运行宿主或插件启动绑定包测试。
- 涉及前端可观察接口联动时，必须按`.agents/rules/testing.md`补充或更新 E2E。
- 涉及接口文档本地化时，必须运行对应`i18n`或`apidoc`治理验证。
- 涉及高频或高数据量接口时，必须通过测试、静态检索或审查记录验证接口契约不会诱导前端瀑布式调用或后端`N+1`查询。

## 审查要求

- 审查必须确认 HTTP 方法、URL 资源语义和副作用语义一致。
- 审查必须确认 DTO 时间字段满足 Unix 毫秒时间戳契约。
- 审查必须确认`g.Meta`、`dc`、`eg`、`permission`等标签完整准确。
- 审查必须确认查询、删除、更新和创建没有使用错误 HTTP 方法。
- 审查必须确认 API 变更的 Go 编译门禁和必要启动绑定测试已执行。
- 审查必须确认高频或高数据量接口具备分页、上限、批量获取或异步处理边界，且响应结构不会诱导前端逐项补查。
