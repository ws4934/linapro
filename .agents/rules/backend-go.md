# 后端 Go 规则

## 适用范围

本规则约束宿主和源码插件的 Go 后端代码、Controller、Middleware、Service、插件宿主服务适配器、WASM host service、公共组件、运行时依赖、错误处理、日志、注释、测试、接口性能、数据库访问频次和 Go 编译门禁。

## 技能和生成代码要求

- 必须使用`goframe-v2`技能开发后端代码。
- 不能修改通过脚手架工具维护的代码文件，例如`api`层的接口方法定义生成文件、`dao`、`do`、`entity`层代码文件。
- `DAO/DO/Entity`源码文件由`make dao`自动生成，不要手动创建或修改。
- `Controller`源码文件由`make ctrl`生成骨架，在生成的文件中填写业务逻辑。

## 注释要求

- 所有源码必须有注释介绍，包括包注释、文件注释、方法注释、常量注释、变量注释和关键逻辑注释。
- 所有 Go 源码文件都必须在文件顶部增加文件用途注释说明。注释必须说明该文件的功能作用、主要实现逻辑和开发者需要注意的约束。
- 组件说明应写在该组件的主文件中，即与组件同名的主文件，例如`plugin.go`、`config.go`、`file.go`。
- 主文件中的组件注释必须紧贴`package xxx`声明，中间不得有空行。
- 非主文件必须只保留当前文件职责注释，文件注释与`package xxx`之间保留一个空行，禁止在非主文件中重复编写组件级说明。
- `service/`目录下每个组件声明的所有接口方法都必须紧邻方法声明提供注释，说明职责、关键行为、输入参数语义、输出结果、零值语义、可能返回的错误类型和必要约束。

## 显式依赖注入要求

- 宿主与源码插件的`Controller`、`Middleware`、`Service`、插件宿主服务适配器和`WASM host service`必须通过构造函数参数逐项显式接收运行期依赖。
- 禁止在业务构造函数、请求处理路径、插件回调路径或 host service 调用路径中临时调用关键服务的`New()`创建独立服务图。
- 禁止通过`Dependencies`、`Deps`、`Options`等聚合结构体把多个接口对象或服务对象字段整体传递给依赖方。
- 接口型依赖必须在构造函数签名中拆分为独立参数，让依赖新增、删除或替换可以在编译阶段暴露所有未同步调用点。
- 纯值配置，例如字符串、布尔、`time.Duration`和容量阈值，可以使用专门配置结构体，但不得混入接口型运行期依赖。

## 初始化错误处理要求

- 宿主与源码插件的 Controller、Middleware、Service、插件宿主服务适配器、WASM host service 配置入口、源码插件注册/registrar API、回调注册 API、路由/Cron/中间件注册 API 和其他运行时初始化方法，在依赖缺失、配置来源缺失、后端创建失败、注册参数非法或校验失败时必须返回`error`。
- 禁止在这类 API 内部直接使用`panic`处理可预期错误。
- 是否中止进程必须由调用栈最上层入口决定。
- 允许`panic`的边界仅限于名称明确的`Must*`辅助函数、静态打包配置、顶层静态注册入口显式处理错误后的失败退出、进程入口最终兜底以及`recover`后重抛未知 panic。
- 新增允许项必须写入 panic 治理扫描 allowlist 并说明原因。

## 缓存敏感服务要求

- 凡是持有缓存、派生状态、失效观察状态、订阅状态、`session/token`状态、插件`enabled snapshot`、运行时配置快照、权限快照或跨实例协调依赖的组件，必须复用启动期传入的同一服务实例或同一共享后端。
- `cluster.enabled=false`时可以使用本地/SQL单机分支。
- `cluster.enabled=true`时必须使用宿主统一的`cluster.Service`、`coordination.Service`、共享修订号、事件广播、分布式 KV 或分布式锁等机制。
- 禁止在中间件、插件管理、源码插件、动态插件 host service 或普通业务路径中退化为仅当前节点可见的默认实例。
- 任何缓存相关变更还必须读取`.agents/rules/cache-consistency.md`。

## Go 编译门禁

- 任何新增或修改 Go 生产代码的任务，在标记完成和通过`lina-review`前，必须基于当前工作区运行至少覆盖变更包的`go test <changed-package> -count=1`或等价编译烟测。
- 涉及 Controller 构造函数、路由绑定、启动编排或 API 接口签名的变更，还必须运行对应宿主或插件启动绑定包测试。
- 宿主默认使用`cd apps/lina-core && go test ./internal/cmd -count=1`，除非提供更窄且能覆盖路由构造的测试。
- 不得仅依赖`git diff --check`、静态扫描、OpenSpec 校验或历史验证记录认定后端 Go 变更可编译。
- 若某个包测试因外部依赖不可用无法运行，必须至少运行能完成编译的替代命令，并在任务记录和审查结论中说明阻断原因、替代覆盖范围和剩余风险。

## 语言和类型要求

- 后端代码中的时间长度统一使用`time.Duration`。
- 配置文件中的时间长度统一使用带单位的字符串，例如`10s`、`5m`、`1h`，并在配置读取层解析为`time.Duration`。
- 禁止在后端实现源码中硬编码具有枚举语义的字符串值。状态、类型、阶段、动作、执行模式、排序方向、过滤操作符等必须使用 Go 命名类型与常量统一管理。
- 禁止为已导入包的导出常量或变量创建包内别名。
- 禁止使用`_ = var`这类单独赋值语句掩盖未使用参数或局部变量；应删除无用变量，或在函数签名中使用空白标识符。

## Error 和 bizerr 要求

- 禁止忽略任何`error`返回值，禁止使用`_ = someFunc()`、`_, _ = someFunc()`或直接调用后丢弃返回值。
- 所有可能进入 HTTP API、插件调用、源码插件后端接口、WASM host service 或其他调用端响应载荷的业务错误、鉴权错误、参数错误和用户可见失败原因，都必须通过`apps/lina-core/pkg/bizerr`的`NewCode`、`WrapCode`或等价封装创建/包装。
- 各业务模块必须在所属模块的`*_code.go`中集中定义`bizerr.Code`。
- 业务调用点禁止硬编码机器错误码或裸错误文案。
- 直接使用`gerror`、`errors.New`或`fmt.Errorf`只允许用于启动期、测试、内部开发诊断、不会返回给调用端的低层技术错误，或在返回边界前会被立即包装为`bizerr`的 cause。

## 日志要求

- 后端宿主代码以及插件后端代码统一使用项目封装的`logger`组件进行日志输出，组件路径为`apps/lina-core/pkg/logger`。
- 除`pkg/logger`封装实现自身外，其他代码不得直接依赖`g.Log()`。
- 日志调用必须传递调用链上下文`ctx`。
- 只要需要打印日志，就必须将上层传入的`context.Context`沿调用链继续传递到当前方法。
- 禁止在已有业务`ctx`可传递的情况下临时使用`context.Background()`、`context.TODO()`或丢弃`ctx`后打印日志。
- 只有启动期、进程级初始化、测试构造或确实不存在请求上下文的入口，才允许显式使用`context.Background()`。

## 分层和文件组织要求

- `apps/lina-core/pkg/`只用于承载宿主与插件、构建工具链或跨组件复用的稳定公共组件。
- 宿主私有共享代码应放在`internal`下具有明确职责名的包中。
- 禁止新增`internal/util`、`internal/common`、`internal/helper`这类语义模糊的兜底目录。
- 仅服务单一业务组件的辅助逻辑应优先放回该组件目录。
- 需要被`internal`目录外复用时再提升到`pkg/<component>`，并补齐公共契约注释。

## 后端复杂度治理要求

- Go 组件接口必须保持职责单一、命名直观、参数清晰、返回值明确。调用方不应为了完成常规操作理解多层间接对象、复杂选项树或隐式初始化顺序。
- 实现逻辑必须优先采用直接、明确的控制流，使用早返回、窄函数和清晰领域命名降低嵌套层级；禁止通过深层嵌套、隐式状态、反射式分发或多级转发隐藏简单业务逻辑。
- 新增`interface`、`manager`、`provider`、`adapter`、`registry`、`factory`、`options`或`facade`必须有明确当前需求：隔离真实外部依赖、承载稳定跨包契约、支持必要测试替身、收敛治理边界或消除跨多处重复复杂逻辑。
- 仅由单一实现满足、只在单包内部调用、只包装一个函数、只转发参数或仅为命名统一而存在的抽象默认视为过度设计；应优先使用私有函数、明确结构体方法或局部直接实现。

## 导出面与内部实现组织要求

- Go 组件默认保持最小导出面。只有被其他包稳定调用、框架或代码生成器要求、公开 API/插件协议/host service 契约要求的标识符，才允许首字母大写导出。
- 仅在当前包内部使用的函数、方法、类型、接口、常量、变量和结构体字段必须使用首字母小写定义；不得为了测试便利、临时复用、规避循环依赖或临时跨包调用而扩大导出面。
- 测试应优先通过公开契约、同包测试或职责明确的`internal/testutil`覆盖内部行为；禁止仅为测试暴露生产导出标识符。
- `api/` DTO、生成代码、框架路由绑定方法、插件入口、host service、`pluginbridge`协议入口等必须导出的内容属于公开契约，必须保持注释、命名和边界清晰；契约不应夹带内部实现结构、缓存状态、`DAO/DO/Entity`或临时配置类型。
- 所有组件的主文件必须作为组件契约入口，只保留组件级说明、公开类型、公开常量、公开接口、构造函数、轻量校验、编排委托和必要适配；具体实现逻辑不得放在组件的主文件中，必须迁移到同包职责明确的非主文件或`internal/<subcomponent>`子组件中。
- 当组件实现复杂，包含多职责逻辑、跨文件共享状态、解析/编解码、运行时执行、缓存协调、插件桥接、外部资源访问或可独立测试的内部流程时，必须在组件根目录下建立职责明确的`internal/<subcomponent>`子组件封装内部逻辑。与该`internal`目录同级的公开包源文件必须只保留公共契约、构造、轻量校验、编排委托和必要适配。
- `internal/<subcomponent>`子组件自身仍必须保持最小导出面；只导出父组件或同一`internal`授权边界内调用所必需的窄契约，其余实现继续使用首字母小写。子组件命名必须体现职责，禁止使用`util`、`common`、`helper`等兜底包名。

## Service 层要求

- 默认使用`Service`作为组件对外暴露的默认接口名，使用`serviceImpl`作为组件实现的默认结构体名称。
- 每个 service 组件的`Service`接口、`var _ Service = (*serviceImpl)(nil)`断言、`serviceImpl`主结构体和`New()`构造函数，必须统一放在组件主文件中。
- 主文件必须作为组件契约入口，只保留组件级说明、核心类型定义、接口定义、默认实现结构体、编译期接口断言和构造函数。具体实现逻辑必须迁移到同包其他文件中。
- `internal/service/`目录下每个组件源文件必须以组件名作为前缀，使用下划线分割子模块，例如`internal/service/config/config_session.go`。
- 同一 service 组件下不同子模块的业务逻辑必须拆分到独立 Go 文件。
- 第一个参数始终传入`ctx context.Context`。
- 数据库交互必须使用 DO 对象，不使用`g.Map`传递`Data`参数。
- 事务管理使用`dao.Xxx.Transaction()`闭包处理多步操作。
- 数据库操作必须使用跨数据库类型的通用语法，禁止依赖特定数据库内置函数。
- 单列固定排序必须使用`OrderAsc`或`OrderDesc`；多列固定排序必须链式调用这些方法，禁止手工拼接排序字符串。

## 接口性能与数据库访问要求

- 所有 API 和 service 实现必须评估数据库访问次数与输入规模、分页大小、返回行数、树节点数、插件数、权限项数或关联对象数之间的关系。
- 禁止在动态结果集循环中逐行执行数据库查询、逐项调用单对象 service 查询或逐项访问远程插件能力来装配列表、批量详情、导出、聚合统计、树形数据或下拉候选；除非循环上限为固定小常量并在设计或审查记录中说明。
- 关联数据装配必须优先收集业务键或外键后使用批量查询、集合化查询、投影查询、聚合查询、缓存或快照一次性获取，再通过内存映射完成合并，避免`N+1`查询。
- 列表接口必须先在数据库侧完成过滤、排序、分页和数据权限约束，再对当前页结果进行必要的批量装配；禁止先加载大集合到内存后分页或过滤。
- 查询应只读取当前接口需要的字段或稳定投影，禁止为了少量展示字段加载完整实体集合后再丢弃大部分字段。
- 字典、配置、权限快照、插件清单等稳定或派生数据应复用既有缓存、快照或批量读取能力；涉及缓存状态时必须继续遵守`.agents/rules/cache-consistency.md`。
- 导出、批量状态变更、批量删除和批量授权等大数据量动作必须采用分批处理、批量读写和事务边界控制，禁止为每条记录独立开启事务或执行重复查询。
- 跨模块或插件能力在列表、批量和聚合路径中必须使用批量契约或稳定投影契约，禁止循环调用单项详情接口作为常规数据装配方式。

## Controller 层要求

- 所有控制器依赖的 service 必须在控制器结构体中定义为字段，并在`NewV1()`等构造函数中初始化或显式传入。
- 禁止在控制器方法内部临时调用`service.New()`创建实例。
- 每个控制器的`_new.go`文件定义控制器结构体和构造函数，其他文件实现具体接口方法。

## 定时任务要求

- 所有定时任务必须在`service/cron`独立组件中统一管理。
- 禁止在`cmd/`或其他 service 组件中直接编写定时任务逻辑。
- `cron`组件提供统一的`Start(ctx)`入口方法，由`cmd`层一次性调用启动所有定时任务。
- 定时任务具体业务逻辑必须在对应业务模块中实现，`cron`模块只负责任务注册和调度。

## 单元测试要求

- 单元测试必须自包含且顺序无关。
- 每个测试方法必须在自身函数内部闭环完成测试场景、数据、依赖替身和清理逻辑的构造与注册。
- 可以复用 helper/fixture，但必须由当前测试显式调用。
- 禁止把其他测试方法的执行结果、数据库残留、全局状态、副作用或执行顺序作为当前测试通过前提。
- 测试清理必须使用当前测试注册的`defer`、`t.Cleanup`或等价机制完成。
- 高频或高数据量接口的测试应优先覆盖批量装配路径；具备数据库测试替身时应增加查询次数断言，无法断言时必须通过静态检索或审查记录说明`N+1`规避依据。

## 审查要求

- 审查必须确认命中后端 Go 规则的变更已经读取本文件。
- 审查必须确认 Go 编译门禁覆盖当前工作区变更。
- 审查必须确认显式依赖注入、初始化错误返回、bizerr、日志上下文、注释、文件组织、导出面最小化、内部实现封装、复杂度治理、接口性能、数据库访问频次和测试自包含性。
- 审查必须拒绝无当前需求支撑的`interface`、`manager`、`provider`、`adapter`、`registry`、`factory`、`options`、`facade`、深层嵌套流程和转发型抽象。
- 审查必须拒绝无固定上限的动态循环查询、列表逐项详情查询、前端瀑布式补查和未说明依据的`N+1`查询方案。
- 涉及 API、SQL、数据权限、缓存、源码插件、`i18n`或开发工具时，必须继续读取对应规则文件。
