# 前端与 UI 规则

## 适用范围

本规则约束前端页面、路由、组件、表格、表单、弹窗、抽屉、适配器层、权限控制、菜单显示、用户可观察交互、布局、样式和图标使用。

## 目录和路径要求

- 路径别名`#/*`指向`./src/*`。
- 路由模块放在`src/router/routes/modules/`。
- 视图文件放在`src/views/`对应目录。
- API 文件放在`src/api/`对应目录。
- 适配器层放在`src/adapter/`，包括`component`、`form`、`vxe-table`等。
- 全局组件在`src/components/global/`注册，例如表格操作列的`GhostButton`。

## 页面实现要求

- 表格页面使用`useVbenVxeGrid`和`Page`组件。
- 表格操作列使用`ghost-button`和`Popconfirm`。
- 前端样式和交互参考`ruoyi-plus-vben5`项目保持一致。
- 弹窗、抽屉、表单、表格、搜索栏等交互模式必须与参考项目保持一致。
- 布局、间距、字体、颜色等视觉元素必须参考现有项目实现，避免引入不一致视觉语言。

## 组件使用要求

- 表单使用`useVbenForm`。
- 弹窗使用`useVbenModal`。
- 抽屉使用`useVbenDrawer`。
- `RadioGroup`单选项使用`optionType: 'button'`和`buttonStyle: 'solid'`。
- 文件上传使用`Upload.Dragger`。
- 文件下载使用`requestClient.download`方法。
- 操作列的“更多”下拉菜单使用`Dropdown`、`Menu`和`MenuItem`。
- 图标使用`IconifyIcon`组件，图标名使用 Iconify 格式，例如`ant-design:inbox-outlined`。

## 前端复杂度治理要求

- 页面组件、业务组件、组合式函数和适配器接口必须保持职责清晰、属性简洁、事件语义明确，调用方应能通过名称、`props`和`emits`直接理解使用方式。
- 组件实现必须优先采用直接的数据流和局部状态，避免不必要的多层`wrapper`、过宽`props`、复杂插槽协议、全局状态绕行或深层组件嵌套。
- 新增公共组件、适配器、组合式函数或状态模块必须有明确当前复用场景或复杂度收益；只服务单一页面、只包装一两个基础组件或只转发参数的逻辑应优先保留在局部实现中。
- 表格、表单、弹窗和抽屉配置应围绕当前页面职责保持可读；禁止为了预留未来页面差异而引入大而全配置对象、动态渲染协议或难以追踪的条件拼装链。
- 前端 API 适配层应做必要的数据投影、命名转换和错误处理，不得把简单调用包装成多层异步转发链，也不得把后端 DTO 原样扩散到无关组件。

## 弹窗和工作流要求

- 导入弹窗必须包含拖拽上传区域、文件类型提示、下载模板链接和覆盖开关。
- 重置密码弹窗必须包含用户信息展示和密码输入。
- 页面工作流应保证常用操作可发现、可撤销或可确认，错误提示清晰。
- 前端模块禁用时，相关 UI 元素必须完全隐藏，而不是仅禁用或置灰。

## i18n 和测试要求

- 修改用户可见文案、菜单、路由、按钮、表单、表格、提示信息时，必须读取`.agents/rules/i18n.md`。
- 涉及用户可观察行为变化时，必须读取`.agents/rules/testing.md`并新增或更新 E2E 测试。
- 涉及 API 调用契约变化时，必须读取`.agents/rules/api-contract.md`。
