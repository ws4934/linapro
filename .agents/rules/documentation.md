# 文档编写规则

## 适用范围

本规则约束仓库内技术文档、目录级说明文档、OpenSpec 文档、Markdown 格式、链接、表格、代码块、图示以及中英文镜像文档的维护方式。

## 基础要求

- `README.md`等技术文档编写必须遵循`.agents/instructions/markdown-format.instructions.md`。
- 仓库内所有目录级主说明文档统一使用英文`README.md`，并同步提供内容一致的中文镜像`README.zh-CN.md`。
- 新增目录说明文档时，必须在同一次变更中同步创建`README.md`与`README.zh-CN.md`，不允许只维护单语版本。
- 同一目录的中英文说明文档必须表达同一事实，不得出现功能范围、命令、路径、约束或状态不一致。

## Markdown 格式要求

- 专业术语、命令、路径、配置项、代码标识符必须使用 inline code，例如`RuntimeClass`、`containerd`、`GPU`、`AI`。
- 中文段落中，inline code 前后不额外添加空格。
- 中文内容使用全角标点，例如`，`、`；`、`。`。
- 参数、配置项、状态矩阵等结构化内容优先使用表格；表格应保持短小，避免正常阅读时横向滚动。
- 架构图、流程图等可视化内容优先使用`mermaid`。`mermaid`内部需要换行时使用`<br/>`，不要使用`\n`。
- 非`mermaid`代码块若使用框线字符，例如`┌─`、`┐`、`┤`或`│`，内容应使用英文，避免中英文宽度导致对齐失真。
- 禁止使用`---`作为普通段落分隔线。
- 生成大篇幅文档时，应拆分为多个明确任务，避免单次输出过大导致质量下降或上下文溢出。

## OpenSpec 文档语言要求

- 新建迭代文档时，`proposal.md`、`design.md`、`tasks.md`与增量规范的内容语言必须跟随用户输入的上下文语言。
- 用户以中文描述需求或明确要求中文时，OpenSpec 文档使用中文。
- 用户以英文描述需求或明确要求英文时，OpenSpec 文档使用英文。
- 若用户明确指定输出语言，则以该指定为准。
- 同一个活跃 OpenSpec 变更中的文档默认保持同一语言，避免一次迭代内出现中英文混写；除非用户明确要求对当前变更文档执行整体语言切换。

## 验证要求

- 文档变更必须运行适合问题性质的治理验证，例如`openspec validate`、静态检索、文件存在性检查、链接检查、格式检查或审查结论。
- 新增目录级说明文档时，必须检查`README.md`和`README.zh-CN.md`是否同时存在。
- 修改 OpenSpec 文档时，必须运行对应变更的`openspec validate <change> --strict`。

## 审查要求

- 审查必须确认命中文档规则的变更已经读取本文件。
- 审查必须确认中英文镜像文档是否同步维护。
- 审查必须确认 Markdown 结构清晰、链接和路径准确、命令可执行、格式满足`.agents/instructions/markdown-format.instructions.md`。
- 若文档变更确认不涉及目录级说明、OpenSpec 文档或格式治理，审查结论中必须记录无文档治理影响。
