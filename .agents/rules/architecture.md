# 架构设计规则

## 适用范围

本规则约束项目定位、核心宿主边界、模块设计、字典枚举、模块启停、模块间接口契约、接口性能设计、数据库访问频次控制、前后端解耦、业务能力可禁用和通用能力复用方式。

## 项目定位要求

- 项目统一定位是`面向可持续交付的 AI 原生全栈框架`。所有功能设计都必须围绕这一定位展开。
- 默认管理工作台、系统管理模块、用户权限模块等能力属于`LinaPro`提供的默认入口和内建通用能力，不构成项目的唯一产品边界。

## 核心宿主边界要求

- `apps/lina-core`是全栈开发框架的核心宿主服务，负责提供通用模块接口能力、组件能力、系统治理能力与插件扩展能力。
- `apps/lina-core`的设计必须优先保证通用性、稳定性和可复用性，不得与具体管理工作台页面的展示结构、交互细节或前端框架实现强绑定。
- 若需求仅来源于表格列、筛选项、树选择器、路由装配、工作台聚合、下拉选项等工作台展示变化，应优先通过工作台适配接口或前端适配层解决，而不是直接修改`lina-core`核心领域契约、通用`service`语义或存储模型。

## 架构复杂度治理要求

- 模块边界、组件接口和跨模块契约必须以当前稳定职责为中心保持简洁，调用方应能通过契约名称、参数和返回值直接理解能力语义、数据边界和错误语义。
- 架构设计必须优先采用直接调用、明确接口和短调用链；禁止为了预留未知未来变化而引入多层`provider`、`manager`、`facade`、`adapter`、`registry`或配置驱动框架。
- 新增抽象层必须满足至少一个条件：隔离已确认的变化点、承载稳定跨模块契约、消除跨多处重复的复杂逻辑、降低调用方理解成本或收敛安全与治理边界。
- 仅包装单一调用、只做参数透传、制造统一外观、隐藏简单条件判断或把局部逻辑提升为通用能力的抽象，视为过度设计，应拒绝。
- 实现流程应保持直接、明确、低嵌套。复杂业务分支应通过清晰的早返回、窄函数和领域命名拆解表达，不得用深层嵌套、隐式状态机或多级间接分发掩盖简单逻辑。

## 模块功能设计要求

- 业务模块的枚举值都应当使用字典模块维护其对应的字典类型和字典数据，而不是在代码中硬编码枚举值。
- 操作日志中的操作类型、操作结果，登录日志中的登录状态，文件管理的业务场景等都应该使用字典类型维护。
- 具有状态、类型、阶段、动作、执行模式、排序方向、过滤操作符等枚举语义的后端值，还必须遵守`.agents/rules/backend-go.md`中的命名类型和常量要求。
- 涉及用户可见字典文案时，还必须读取`.agents/rules/i18n.md`。

## 模块解耦设计原则

- 所有前后端模块必须采用解耦设计，业务模块支持按需启用和禁用。
- 禁用某模块后，所有依赖该模块的功能必须自动降级或隐藏，不能出现报错或空白区域。
- 当一个模块被禁用时，前端所有涉及该模块的 UI 元素必须完全隐藏，而非仅禁用或置灰。范围包括菜单项、表单字段、表格列、搜索条件、按钮等。
- 后端服务间的依赖应通过接口或可选引用实现，避免硬依赖。
- 当被依赖模块禁用时，相关字段返回零值或忽略即可，不应抛出错误，除非规范明确要求拒绝访问。
- 模块禁用仅影响功能和展示层，不应删除或破坏已有数据。
- 重新启用模块后，历史数据应能正常恢复使用。

## 模块接口契约要求

- 所有模块之间的运行时调用必须通过明确的接口契约、稳定能力接缝或受治理协议完成，不得直接调用其他模块的内部实现方法，也不得直接访问其他模块的内部数据结构。
- 接口契约可以是后端公开`interface`、`pkg`稳定能力接缝、`HTTP API`与`DTO`、插件`hostServices`、`pluginbridge`、版本化 host service、事件、命令或查询契约；契约必须有清晰 owner，并说明调用职责、数据边界、错误语义和替换策略。
- 跨模块契约返回值必须使用契约自有的投影结构、值对象或`DTO`，不得向调用方泄漏被调用模块的`DAO`、`DO`、`Entity`、私有配置、缓存快照、运行时状态或内部模型。
- 跨模块依赖必须通过构造函数显式注入接口、可选能力引用、registrar、provider、adapter 或受治理能力发现机制接入，不得通过全局服务定位、临时`New()`内部实现、私有注册表或绕过启动装配的方式耦合其他模块。
- 即使语言或目录结构允许访问，被调用模块的`internal/service`实现类型、`internal/dao`、`internal/model`、未发布缓存结构、未发布配置结构和私有 helper 也不得作为其他模块的生产调用入口。
- 启动装配、registrar/provider 适配、数据库迁移、代码生成、测试 fixture 和治理扫描可以在受控边界内接触多个模块，但必须限定在装配、适配、验证或测试职责中；若触及内部实现，设计或审查结论必须说明原因、范围和不进入业务运行路径的约束。

## 接口性能设计要求

- 功能设计阶段必须把接口性能作为契约约束，尤其是列表、详情批量、导出、聚合统计、树形数据、下拉候选、工作台聚合、插件资源扫描等会随数据量增长的场景。
- 数据装配成本必须随页大小、节点数、插件数、权限项数或关联对象数保持有界且可解释，禁止默认接受`N+1`查询、前端瀑布式调用或逐项同步远程调用作为常规实现方案。
- 跨模块数据装配应优先通过批量契约、投影查询、缓存、快照或事件驱动派生数据完成，不得为了复用单项详情接口而在列表、批量或聚合路径中循环调用其他模块查询。
- `apps/lina-core`的通用接口设计必须优先满足框架级复用和高数据量装配需求，不得只围绕当前工作台页面的少量展示字段设计单点查询契约。

## 设计记录要求

- OpenSpec 设计必须说明新增或修改模块是否属于宿主通用能力、源码插件能力、工作台适配层能力或前端展示适配。
- 设计必须说明组件接口和实现路径的复杂度判断；涉及新增抽象、适配层、扩展点或配置模型时，必须说明为什么局部直接实现不足，以及该抽象如何降低当前复杂度。
- 涉及模块启停时，设计必须说明禁用后的前端隐藏策略、后端降级策略、数据保留策略和重新启用恢复路径。
- 涉及枚举语义时，设计必须说明字典类型、字典数据来源、运行时展示方式和`i18n`影响。
- 涉及跨模块调用或接口契约时，设计必须说明契约 owner、调用方、被调用方、契约形态、降级策略、数据权限、缓存一致性、`i18n`和错误语义影响，并明确不会暴露被调用模块内部结构。
- 涉及高频或高数据量接口时，设计必须说明数据规模假设、数据库访问次数如何随输入规模变化、`N+1`规避方案、必要索引、缓存或快照策略以及验证方式。

## 验证要求

- 涉及模块启停或前端隐藏时，必须使用单元测试、E2E 或治理验证覆盖启用和禁用两种状态。
- 涉及后端模块降级时，必须验证依赖模块禁用后不会出现 500、空白页或不可恢复错误。
- 涉及跨模块接口契约变更时，必须使用单元测试、编译门禁、静态检索、导入边界检查或治理审查验证调用方只依赖契约，不依赖被调用模块内部实现或内部数据结构。
- 涉及高频或高数据量接口时，必须通过单元测试、集成测试、查询次数断言、静态检索或审查记录验证不存在`N+1`查询和不必要的前端瀑布式调用。
- 纯治理文档变更可以使用`openspec validate`、静态检索、文件检查或审查结论作为验证证据。

## 审查要求

- 审查必须确认变更没有因工作台展示需求直接污染`lina-core`核心领域契约。
- 审查必须确认组件接口、模块契约和实现路径保持清晰简洁，新增抽象层、适配层、扩展点或配置模型均有当前需求支撑并降低真实复杂度。
- 审查必须确认跨模块调用均通过接口契约、稳定能力接缝或受治理协议完成，没有直接调用其他模块内部方法或访问其他模块内部数据结构。
- 审查必须确认高频或高数据量接口已经评估数据库访问频次、批量装配方式、索引或缓存策略，并拒绝无约束的`N+1`查询方案。
- 审查必须确认模块启停、前端隐藏、后端降级和数据完整性策略已记录并验证。
- 审查必须确认具有枚举语义的业务值没有绕过字典治理。
- 涉及数据操作、缓存、插件或`i18n`时，审查必须继续读取对应规则文件。
