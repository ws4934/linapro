## Context

普通文件上传使用 `LocalStorage.Put` 生成相对存储路径，并将该路径写入 `sys_file.path`。当前规则是 `t/<tenantId>/<yyyy>/<MM>/<generatedName>`，其中 `t` 是 tenant 的缩写。访问文件时，系统从 URL 中取相对路径，再查询 `sys_file.path` 并通过存储后端读取文件，因此历史记录是否可访问取决于数据库中的路径，而不是固定的新路径规则。

## Goals / Non-Goals

**Goals:**

- 新上传文件去掉额外的 `t` 目录层。
- 保留租户 ID 目录，继续支持物理文件按租户分区组织。
- 不迁移历史 `t/...` 文件，确保旧记录继续通过既有数据库路径访问。
- 用单元测试覆盖新路径规则和旧路径兼容边界。

**Non-Goals:**

- 不删除租户 ID 目录。
- 不修改数据库 schema、DAO、实体或现有文件记录。
- 不调整插件 host storage service 的路径规则。
- 不改变 `/api/v1/uploads/*path` 访问接口、权限模型或数据权限边界。

## Decisions

### Decision 1: 只移除字面量 `t` 前缀

新路径采用 `<tenantId>/<yyyy>/<MM>/<filename>`。这样可以满足去掉 `t` 目录的需求，同时保留租户隔离和运维定位所需的目录层级。

替代方案是完全去掉租户目录，只保留 `<yyyy>/<MM>/<filename>`。该方案会降低物理目录层面的租户可识别性，并让后续按租户清理或迁移更困难，因此不采用。

### Decision 2: 不迁移历史文件

历史 `sys_file.path` 已经保存完整相对路径，旧文件可以继续通过 `t/...` 路径访问。迁移历史文件会涉及文件系统移动、数据库批量更新、失败回滚和重复 hash 记录处理，超出本次低风险路径规则调整范围。

### Decision 3: 保持 hash 去重复用现状

上传流程仍按 `hash + tenant_id` 查重。若上传内容与历史文件相同，系统会继续复用已有历史路径；只有真正写入新物理文件时才采用新路径规则。这样避免重复存储同一文件，也避免为了路径外观破坏现有去重语义。

## Risks / Trade-offs

- 旧文件和新文件路径格式会在一段时间内共存。缓解方式：明确记录兼容策略，并通过测试确认旧 `t/...` 路径仍能访问。
- 相同内容重复上传可能仍返回旧 `t/...` 路径。缓解方式：将该行为记录为非迁移策略的一部分，避免误判为 bug。
- 注释和 API 示例可能与实现不一致。缓解方式：同步更新相关示例并运行校验。

## Required Assessments

- i18n：不新增、修改或删除用户可见文案；不需要维护前端运行时语言包、插件 manifest/i18n 或 apidoc i18n。
- 缓存一致性：不新增或修改缓存；文件访问仍以数据库 `sys_file.path` 为权威路径，分布式环境下没有新增缓存一致性风险。
- 数据权限：不新增或修改数据操作接口；上传记录仍写入当前租户，读取仍通过现有元数据和租户过滤校验。
- 开发工具跨平台：不新增或修改开发工具、脚本、CI 入口或默认开发命令。
