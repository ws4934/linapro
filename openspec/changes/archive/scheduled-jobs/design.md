## Context

### Current State

- `apps/lina-core/internal/service/cron` exists, hosting host-side code-embedded tasks: `session-cleanup`, `servermon-collector`, `servermon-cleanup`, `access-topology-sync`, `runtime-param-sync`, and plugin-level cron injected by `pluginSvc.RegisterCrons`.
- The underlying scheduler is GoFrame's `gcron`; leader election comes from `service/cluster`; distributed lock capabilities come from `service/locker` and `service/hostlock`.
- All built-in tasks are registered by code at process startup, with no user-visible entry point and no unified persistent projection.

The startup path initially used `sys_job` as both the governance projection and the execution registration source for built-in jobs. This created duplicate registration paths and forced the scheduler to use an idempotency patch for same-name gcron entries. The boundary was refined so that built-in jobs are executed from host code or plugin declarations, and `sys_job.is_builtin=1` only stores console display data, log linkage, audit snapshots, and governance state.

### Requirements Overview

Add a "Scheduled Job Management" subsystem for operations users, covering CRUD, groups, enable/disable, execution logs, count/concurrency policies, manual trigger and cancellation, with support for:
- **Handler mode**: Calls host- or plugin-registered named handlers; parameters validated via JSON Schema.
- **Shell mode**: Executes `/bin/sh -c` multi-line scripts with independent permissions and global switch.

### Constraints

- Must reuse existing `service/cron` and `service/cluster`; no second scheduler.
- Backend follows `GoFrame v2` conventions; DAO/DO/Entity generated by `make dao`.
- Frontend strictly follows `Vben5 + Ant Design Vue` and must match `ruoyi-plus-vben5` interaction style.
- Enumerations go through the dictionary module, not hardcoded literals.
- Database operations use cross-database compatible SQL, avoiding MySQL-specific functions.
- Time durations use `time.Duration`; config files use `"10s"/"5m"/"1h"` string format.

## Goals / Non-Goals

**Goals:**

1. Provide a complete user-manageable scheduled job system (CRUD, groups, enable/disable, logs, cleanup policies).
2. Unified display of user-created jobs, host built-in jobs, and plugin built-in jobs, avoiding real running scheduled jobs being invisible in the management page.
3. Reuse gcron on top of existing `service/cron`, supporting runtime CRUD through dynamic registration.
4. Plugin handlers automatically go online/offline during plugin enable/disable events, with associated jobs cascading state without data loss.
5. Shell mode implements multi-layer security: independent permission point + global switch + audit log + process group kill + output truncation.
6. Correctly distinguish `master-only` and `all-node` scheduling scope in cluster mode, following existing `cluster.IsPrimary()` guard.
7. Source-code registered jobs treat code as the single source of truth, projecting into `sys_job` for display and trigger only; administrators cannot modify definitions through public UI/API.
8. Built-in jobs are registered directly from host code definitions or plugin declarations during startup; persistent scheduler loading only handles user-defined jobs (`is_builtin=0`).
9. UI follows Vben5 component specifications; user creation entry only exposes Shell task forms.

**Non-Goals:**

1. No "global singleton" third scheduling mode (this iteration only master-only / all-node).
2. No queuing mode (parallel exceeding `max_concurrency` skips rather than queues).
3. No Windows shell execution (production is Linux; development Windows defaults shell mode to disabled with a prompt).
4. No cron expression visual editor (this iteration only text input + next execution time preview).
5. No handler parameter custom component library for frontend, only mapping JSON Schema to Vben built-in controls (input, number, switch, dropdown, date, etc.).
6. No "task dependency / DAG" semantics in this iteration.
7. No handler script sandbox (Wasm/JS) — plugin handlers use existing Wasm plugin system.
8. No public UI/API entry for creating Handler tasks in this iteration; such tasks only allow source-code registration.
9. No new database tables or external scheduler dependencies for the execution boundary refinement.
10. No change to CRUD, enable/disable, manual trigger, or shell-job permission semantics for user-defined jobs.

## Decisions

### D1. Task type: single table + `task_type` field, not two tables

**Decision**: `sys_job` single table, distinguished by `task_type in {handler, shell}`, with type-specific fields filled as needed (handler uses `handler_ref` + `params`, shell uses `shell_cmd` + `work_dir` + `env`).

**Rationale**: Both types share many fields (group, name, cron, scope, concurrency, executed_count, status, is_builtin, log_retention); splitting into two tables would cause repeated joins in query/list scenarios. Single-table design enables unified scheduler handling. The downside is some columns being permanently null for one type, which is acceptable sparsity.

**Alternative**: Main table stores only common fields, `sys_job_handler` / `sys_job_shell` store differing fields — rejected, query complexity far outweighs benefits.

### D2. Scheduler reuses gcron, persistent state loaded from database

**Decision**:
- At startup, `service/cron/cron.go` adds a one-time operation: `loadPersistentJobs(ctx)` reads from `sys_job where status=enabled and is_builtin=0` and registers to gcron.
- Built-in jobs are synchronized separately: `cron.syncBuiltinScheduledJobs` upserts `sys_job` projection rows first, then uses the resulting `sys_job.id` to register gcron entries from code definitions or plugin declarations. Execution fields come from the in-memory model, not from a later scan of built-in rows.
- Job CRUD goes through the `JobScheduler` component at the controller/service layer, which handles atomic `gcron.Remove + gcron.Add` refresh.
- The scheduler callback is a thin wrapper `runJob(jobID)`, which reads the latest job config from the database on each tick (hitting a small LRU cache to avoid high-frequency DB pressure), then dispatches to handler or shell executor.

**Rationale**: No custom scheduler; preserving gcron's maturity. The wrapper reads config from DB to solve the "CRUD modifies parameters but scheduler still uses old snapshot" problem; cache invalidation happens proactively at CRUD completion. The alternative of letting `LoadAndRegister` scan all `enabled` jobs and using `gcron.Remove` or same-name overwrite for idempotency was rejected because it keeps the wrong boundary: built-in execution is still driven backward from the table and startup still needs extra queries and duplicate-registration protection.

**Alternatives**:
- Full in-memory registry, CRUD directly modifies memory — rejected, process restart causes drift with DB, no consensus across nodes.
- Read DB on every tick without cache — rejected, high-frequency tasks may pressure DB; cache + proactive invalidation is more reasonable.

### D3. Cluster scheduling scope: reuse existing two tiers, no global singleton

**Decision**: `scope in {master_only, all_node}`.
- `master_only`: All nodes register gcron, trigger checks `cluster.IsPrimary()` first, non-primary nodes return directly.
- `all_node`: Each node registers and executes independently.

**Rationale**: Existing `service/cron` already implements these two tiers; migration cost is lowest. Global singleton has higher understanding cost for users, and typical user tasks (cleanup, sync, health probe) are adequately served by master-only.

**Risk**: During primary switch, a tick may be skipped (new primary hasn't completed leader claim); logs record `status=skipped_not_primary`, observable by operations. Tasks needing strong guarantees should have handlers implement idempotency (external state recording).

### D4. Concurrency policy: singleton / parallel + max_concurrency

**Decision**: `concurrency in {singleton, parallel}`.
- `singleton`: Per-node singleton (`gcron.SetSingleton(true)` or equivalent semantics), skips new tick when already running, logs `status=skipped_singleton`.
- `parallel`: Per-node soft cap of `max_concurrency`; skips new tick when at limit, logs `status=skipped_max_concurrency`.

**Rationale**: Simple and observable. "Global singleton" requires distributed locks, adding complexity; not a goal for this iteration.

**Alternative**: Queue when at limit — rejected, queues introduce secondary complexity (persistence, restart loss, starvation).

### D5. Handler registration: host built-in + plugin subscription, unified through `HandlerRegistry`

**Decision**: New `service/jobhandler` component, defining `Registry` interface:

```go
type Registry interface {
    Register(ref string, def HandlerDef) error  // ref like host:xxx or plugin:<pluginID>/<name>
    Unregister(ref string)
    Lookup(ref string) (HandlerDef, bool)
    List(ctx context.Context) []HandlerInfo  // for UI dropdown
}

type HandlerDef struct {
    Ref          string                     // unique identifier
    DisplayName  string                     // UI display name
    Description  string                     // description
    ParamsSchema string                     // JSON Schema text
    Source       HandlerSource              // host | plugin
    PluginID     string                     // filled when from plugin
    Invoke       func(ctx context.Context, params json.RawMessage) (result any, err error)
}
```

- Host startup phase calls `Registry.Register("host:clean-session-logs", ...)` to register built-in handlers.
- `service/plugin` enable/disable/uninstall success paths synchronize handler availability through explicitly injected lifecycle observers calling `Registry.Register` / `Unregister`, rather than relying on loose asynchronous event buses.
- `Registry.Unregister(ref)` triggers `JobScheduler` cascade: all jobs with `handler_ref=<ref>` and `status=enabled` are set to `paused_by_plugin`, UI highlights red; re-`Register` automatically restores to `enabled` (only restoring jobs with `stop_reason=plugin_unavailable`).

**Rationale**: Registry is the single source of truth; task scheduling, UI dropdown, and parameter validation all go through it; cascade logic is centralized, avoiding scatter across callers.

**Schema boundary**:
- `ParamsSchema` uses `JSON Schema draft-07` restricted scalar subset; root must be `type=object`.
- Only supports `properties / required / description / default / enum / format` and field types `string / integer / number / boolean`.
- `format` only supports `date / date-time / textarea` for frontend mapping in this iteration.
- Explicitly rejects `array`, nested `object`, `$ref`, `allOf`, `anyOf`, `oneOf`, `not`, `patternProperties` and other keywords beyond this iteration's form mapping capability.
- Backend parameter validation must use a `draft-07`-compatible schema validator; frontend dynamic forms do deterministic mapping per the same subset.

### D6. Shell security layers

**Decision**: Shell tasks introduce three-layer protection:

```
L1 Permission     system:job:shell permission point, default only admin has
L2 Global switch   cron.shell.enabled = true/false, operations can take over at any time
L3 Audit           shell task create/modify/trigger/cancel reuse host HTTP OperLog middleware,
                   writing exactly one oper_log record per operation
```

**Runtime**:
- Fixed `/bin/sh -c <shell_cmd>` launches `exec.CommandContext`.
- `SysProcAttr.Setpgid = true` (Unix), timeout/manual termination uses `kill -- -pgid`.
- `work_dir` empty uses host process working directory; non-empty validates directory exists + is not root path.
- `env` is a task-level KV map, persisted as plaintext JSON in `sys_job.env`; UI editing masks existing values by default; audit logs do not record raw `env` payloads; execution overlays on top of host process environment (task-level overrides process-level).
- `stdout` / `stderr` each use `io.LimitReader` truncating at first 64KB, appending `...[truncated]` marker on overflow.
- `timeout` is a required field (seconds), range `[1, 86400]`, `ctx, cancel := context.WithTimeout(parentCtx, timeout)`.

**Audit implementation constraints**:
- Shell create/modify/trigger/cancel HTTP interfaces continue through existing `middleware.OperLog`.
- Controllers fill audit semantics through `operLog` meta tags and response payloads (e.g., trigger returns `log_id`).
- Host audit parameter desensitization rules extend to shell request body `env` field, preserving only key names or writing `[REDACTED]`.
- Implementation must not write a second semantically duplicate `sys_oper_log` record in the same request chain.

**Windows handling**: Does not disable compilation; at runtime if `runtime.GOOS=="windows"` then `cron.shell.enabled` is forcibly treated as false, UI shows "Current platform does not support shell mode".

### D7. Built-in job execution boundary

**Decision**: Built-in jobs (`is_builtin=1`) are registered directly from host code definitions or plugin declarations. `sys_job.is_builtin=1` rows serve only as governance projections for console display, log linkage, audit snapshots, and state governance. They MUST NOT be used as the registration source for startup persistent scanning.

**Key behaviors**:
- `cron.syncBuiltinScheduledJobs` synchronizes the `sys_job` projection first, then uses the resulting `sys_job.id` to register the corresponding gcron entry. Cron, scope, concurrency, timeout, handler reference, and other execution fields come from the current in-memory model converted from code or plugin declarations.
- `jobmgmt/internal/scheduler.LoadAndRegister` queries only `is_builtin=0 AND status=enabled`. CRUD, enable/disable, and edit refresh paths continue to serve user-defined jobs.
- Manual trigger remains available for executable built-in jobs but uses declaration-owned execution data. If the plugin handler is unavailable or the job is `paused_by_plugin`, the trigger returns handler-unavailable semantics.
- Delivery SQL does NOT write initialization seed data for built-in jobs into `sys_job`; projections are upserted from code and plugin declarations at runtime.

**Rationale**: Only by projecting all real running tasks into the same task table can the management page avoid the split state of "page shows one set, process actually runs another." Keeping built-in jobs out of persistent loading keeps the persistent scheduler responsibility clear: restore user jobs from user-owned persistent data. Built-in jobs belong to host and plugin runtime capabilities.

**Alternative considered**: Let `LoadAndRegister` continue scanning all `enabled` jobs and use `gcron.Remove` or same-name overwrite for idempotency — rejected, keeps the wrong boundary because built-in execution is still driven backward from the table.

### D7A. Source-code registered jobs: fully read-only + triggerable

**Decision**: Source-code registered jobs visible through the public UI/API are uniformly treated as "read-only definitions":
- No editing, deleting, enabling/disabling, or resetting execution count allowed;
- View details, view logs, and manual trigger once are allowed;
- `status` changes come only from system runtime (e.g., plugin unavailable auto-pause), not from manual modification.

**Rationale**: These jobs' definitions come from host or plugin source code; their safety and correctness should be guaranteed by code review and release processes, not by runtime manual rewriting. Manual trigger is preserved for troubleshooting and verification scenarios.

### D7B. Source-code registered jobs projected through code into `sys_job`

**Decision**:
- Host built-in cron and plugin built-in cron no longer bypass `sys_job` to directly register raw entries into `gcron`.
- The system collects source-code registered job definitions during the startup phase and upserts them into `sys_job`.
- Scheduling execution returns to the persistent scheduler `jobmgmt/internal/scheduler`, ensuring list, detail, logs, immediate execution, and runtime are fully consistent.
- User-created jobs continue to write `sys_job` directly, but only `shell` type is allowed.

### D7C. Dynamic plugin scheduled jobs registered through independent cron host service code, not merged into runtime host service

**Decision**:
- Dynamic plugins register built-in scheduled jobs through an independent `cron` host service in guest code, using `pluginbridge.Cron().Register(...)` to submit `CronContract`.
- The host executes the preserved guest registration entry (`RegisterCrons` / `/register-crons`) during dynamic plugin scan/install/enable, projecting collected `CronContract` as `plugin:<pluginID>/cron:<name>` built-in jobs, continuing to reuse the unified `sys_job`, handler registry, logs, and immediate execution paths.
- Guest-side `runtime` host service continues to only handle runtime logs, status, and lightweight information queries.
- If future plugin-side job governance capabilities are needed, they should fall into an independent `cron`/`job` host service, not continue to be stacked into the `runtime` service.

**Rationale**: Scheduled job governance maintains independent boundaries, does not stack declaration capabilities into the `runtime` service, while allowing dynamic plugins to maintain declaration logic centrally through code like registering data sources.

### D8. Log cleanup: global default + task-level override + built-in cleanup task

**Decision**:
- System parameter adds `cron.log.retention` (default `{mode: days, value: 30}`).
- Task-level `log_retention_override` (JSON, `{mode: days|count|none, value: N}`), null means follow global.
- New built-in task `host:cleanup-job-logs`, triggered daily at midnight, scans `sys_job_log` and cleans by policy. This task is registered through host source code and projected into `sys_job` during service startup; delivery SQL does NOT write seed data for it.

**Rationale**: Cleanup itself is a scheduled job; using this subsystem for bootstrapping is cleaner; `mode=none` means no cleanup (for scenarios where users are willing to bear storage costs).

### D9. Manual trigger and manual termination semantics

**Decision**:
- **Manual trigger**: UI "Run Now" button -> confirmation modal -> `POST /job/{id}/trigger` -> scheduler starts one execution, `trigger=manual`, **not counted in `executed_count`**. The confirmation modal uses execution-specific styling rather than delete styling.
- **Scheduled trigger counting**: Executions triggered normally by the scheduler (`trigger=cron`) count toward `executed_count` regardless of whether `max_executions` is `0`; `max_executions=0` only means "no limit", not "don't count".
- **Manual termination**: On a running log row -> `POST /job/log/{id}/cancel` -> scheduler finds the corresponding goroutine, calls `cancel()`. Handler must respond to `ctx.Done()`; Shell execution kills the process group.
- After termination, log status = `cancelled`, with `end_at` recorded.

### D10. Manual termination and handler specification contract

**Decision**: All handlers (including plugins) must implement:
1. Accept `ctx context.Context` as first argument; long-blocking operations must pass `ctx` through to downstream (HTTP / DB / lock, etc.).
2. Periodically check `ctx.Done()`; upon receiving cancellation signal, exit promptly and return `ctx.Err()`.
3. Cleanup logic uses `defer`, ensuring cleanup runs on cancellation paths.

Handlers violating this contract will, during "manual termination", show "log already marked cancelled but goroutine continues consuming resources"; operations can observe and troubleshoot corresponding handler code.

### D11. RESTful interface design

**Decision**:

```
GET    /job                 List (supports group/status/keyword/type filtering)
GET    /job/{id}            Detail
GET    /job/cron-preview    Cron expression preview (next 5 trigger times)
POST   /job                 Create
PUT    /job/{id}            Update
DELETE /job                 Batch delete
POST   /job/{id}/trigger    Manual trigger once
POST   /job/log/{id}/cancel Manual terminate running instance
PUT    /job/{id}/status     Enable/disable
POST   /job/{id}/reset      Reset executed_count

GET    /job-group           List
POST   /job-group           Create
PUT    /job-group/{id}      Update
DELETE /job-group           Batch delete (default group refused)

GET    /job/log             List (filtered by job/time/status)
GET    /job/log/{id}        Detail (includes full result_json)
DELETE /job/log             Clear (by job or all)

GET    /job/handler         Handler registry list (for dropdown)
GET    /job/handler/{ref}   Handler detail (includes ParamsSchema)
```

**Code organization**: DTO and controllers split by resource into `job / jobgroup / joblog / jobhandler` four directory groups. `/job/log/{id}/cancel` is implemented by the `joblog` controller; the `job` base controller only handles task CRUD, status switching, manual trigger, reset count, and cron preview.

**Public interface boundary**:
- `POST /job` and `PUT /job/{id}` only accept user-created Shell tasks.
- Source-code registered jobs are displayed through the `/job` resource, but their definitions sync through internal projection flows, not through public create/edit interfaces.
- `PUT /job/{id}/status` and `POST /job/{id}/reset` return read-only errors for source-code registered jobs.

**Permission matrix**:
- Menu permissions: `system:job:list`, `system:jobgroup:list`, `system:joblog:list`
- Task buttons: `system:job:add / edit / remove / status / trigger / reset`
- Group buttons: `system:jobgroup:add / edit / remove`
- Log buttons: `system:joblog:remove / cancel`
- Shell additional permission: `system:job:shell`, combined with corresponding `system:job:add / edit / trigger`; if terminating a shell task instance, also combined with `system:joblog:cancel`

## Risks / Trade-offs

- **gcron dynamic registration race condition**: When CRUD and scheduler tick run concurrently, the window between `Remove + Add` may hit a tick, causing an extra trigger or a missed trigger. Mitigation: per-job mutex lock in `JobScheduler` for `Remove/Add` (`sync.Map` per-job mutex), and clear LRU cache before add.
- **Primary switch moment master-only task tick loss**: Before new primary completes leader claim, tick may be skipped. Mitigation: logs clearly record `status=skipped_not_primary`, visible to operations; tasks needing strong guarantees should have handlers implement idempotency.
- **Handler goroutine leak**: When handler does not respond to `ctx.Done()`, "termination" only marks the log, goroutine still runs. Mitigation: specs clearly state contract, review enforces; future may consider panic after timeout if goroutine cannot be terminated (not in this iteration).
- **Shell mode abuse risk**: Even with permission point and switch, administrators can still execute arbitrary commands. Mitigation: global switch defaults to off, needs operations to actively enable; permission point defaults to admin only; all shell operations go to audit log; shell_cmd and work_dir fields display warning colors.
- **Plugin disable causing mass job cascade**: When plugin has 100 jobs, one disable produces 100 `paused_by_plugin` records and logs. Mitigation: `Unregister` cascade uses batch UPDATE + single audit log recording affected scope.
- **Seed upgrade conflicting with user changes**: New version built-in task changed scope, old task exists but scope is a locked field. Mitigation: when `seed_version` is lower, forcibly overwrite locked fields; open fields are never overwritten; operations can explicitly restore through "reset to default".
- **Built-in job registration needs `sys_job.id` for log linkage**: Mitigation: upsert the projection first and return or reload the projection before runtime registration.
- **Plugin lifecycle and cron registration order creating short unavailable window**: Mitigation: keep lifecycle callbacks in the same request chain and complete handler registration, projection sync, and scheduler refresh together.
- **Manual trigger code reading `sys_job` being misread as table-driven execution**: Mitigation: specs and code comments clarify that `sys_job` is only a governance projection and log snapshot, while executability is determined by the handler registry and declaration index.
- **Removing the `gcron.Remove` startup idempotency patch exposes real duplicate registrations again**: Mitigation: reposition duplicate-registration tests to verify that built-in jobs do not enter `LoadAndRegister`, while preserving explicit remove-then-register behavior for CRUD refresh.
- **result_json causing overly wide rows**: Shell output 64KB * 2 paths, extreme case 128KB per row. Mitigation: MySQL uses `longtext`; frontend query list does not include `result_json`, detail API returns it.
- **Task migration on group deletion**: Deleting group migrates tasks to default group. Mitigation: clearly irreversible; UI shows confirmation dialog explaining impact scope.
- **Timezone drift**: When multi-node servers have inconsistent TZ, tasks without timezone have inconsistent trigger points. Mitigation: `timezone` is required, seed defaults to `Asia/Shanghai`, task form defaults to server TZ.

## Migration Plan

### Deployment Steps

1. After merging PR, execute `make init` to trigger `manifest/sql/<NNN>-scheduled-job-management.sql` migration: create three new tables, insert default group, insert system parameters `cron.shell.enabled=false` and `cron.log.retention`, register permission points and menu items, insert dictionaries. Note: built-in job rows are NOT seeded into `sys_job`.
2. Frontend build and deploy, gradually enable `system:job:*` menu items.
3. On backend restart, `service/cron/cron.go`'s `Start(ctx)` calls `jobmgmtScheduler.LoadAndRegister(ctx)` to load `sys_job where status=enabled and is_builtin=0` and register to gcron; built-in jobs are registered from code/plugin declarations through `cron.syncBuiltinScheduledJobs`. Existing code-embedded cron continues through the old path, no interference.
4. Operations enables `cron.shell.enabled=true` on the system parameters page (if shell capability is needed), and grants `system:job:shell` permission to roles that need to execute shell tasks.

### Rollback Strategy

- Backend rollback: Restore code version; data tables preserved (three new tables not referenced by old version), old version no longer loads tasks after restart.
- Frontend rollback: Hide menu entries.
- Data rollback: This change only adds tables and seed; no destructive DDL; no data rollback needed.
- Emergency disable: Set `cron.shell.enabled=false` on system parameters page; shell tasks are immediately blocked.

## Clarifications Locked

1. `env` is stored as plaintext JSON in `sys_job.env` in this iteration; UI editing masks existing values by default; audit logs do not record raw `env` payloads.
2. `timeout_seconds` is a shared public field for both `handler` and `shell` task types; the handler registry no longer has an additional default timeout override chain.
3. `max_concurrency` in `all_node` mode is implemented as "per-node cap".
4. Cron preview interface is fixed as `GET /job/cron-preview?expr=...&tz=...`, provided by the `job` resource.
5. Built-in job delivery SQL does NOT write initialization seed data into `sys_job`; projections are upserted from code and plugin declarations at runtime.
6. The `host:cleanup-job-logs` built-in cleanup task is registered through host source code and projected into `sys_job` during service startup, not through delivery SQL seed.
7. `cron.shell.enabled` defaults to `true`; platform-level unsupported-shell protection can still disable shell jobs safely.
