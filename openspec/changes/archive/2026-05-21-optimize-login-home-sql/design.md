## Context

`temp/login.log` 的统计结果：

- 共 152 条 SQL，累计日志内 DB 时间约 193ms。
- 请求 ID 共 10 个，说明登录后首页触发了多接口并发加载。
- `sys_plugin_release` 查询 85 次，其中按 `plugin_id + release_version` 查询 45 次，按 `id` 查询 39 次。
- `sys_plugin` 查询 16 次。
- `sys_online_session` 查询 18 次，另有 6 次 `last_active_time` 条件更新。
- GoFrame 表结构元数据查询 7 次，约 89ms，主要属于首次访问热点表的冷启动成本。

## Goals / Non-Goals

**Goals:**

- 减少登录后首页运行期重复 SQL，优先治理插件 release 重复读取和会话校验多次 `COUNT`。
- 保持强制下线、JWT revoke、会话超时、租户隔离和插件状态变更语义不变。
- 对行为变化补充自动化测试。
- 明确缓存一致性边界，避免引入只在单节点正确的运行期缓存。

**Non-Goals:**

- 不重构登录、菜单、消息、租户或插件管理 API 契约。
- 不修改数据库 schema 或新增索引。
- 不把 GoFrame 首次表结构元数据查询作为本轮主优化目标。
- 不调整前端首屏加载交互，除非后端优化后仍无法满足目标。

## Decisions

### Decision 1: 会话校验改为一次读取再判断

当前 DB-only `TouchOrValidate` 对有效请求执行存在性 `COUNT`、过期 `COUNT`、条件 `UPDATE`、再次 `COUNT`。这会在首屏并发请求中稳定放大为每请求 3 到 4 次在线会话 SQL。

本变更将 DB-only 校验改为读取一条 `sys_online_session` 记录并检查租户和 `last_active_time`，仅在过期时删除，仅在超出写入节流窗口时更新。这样保持语义不变，同时把常规有效请求减少到 1 次 `SELECT`，必要时再 1 次 `UPDATE`。

### Decision 2: 插件 release 读取优先使用请求级或列表级快照

插件 release 表是小表，但它承载插件安装、启用、升级、卸载等状态。为了避免在集群模式下引入跨节点陈旧状态，本轮不直接增加长 TTL 进程级缓存。优先使用请求级快照或一次性批量读取，在单个首页列表投影、运行期构建或管理请求内复用 release 行。

如未来需要进程级缓存，必须接入 `cachecoord` 或等价 revision 机制，并在插件生命周期状态变更后跨实例失效。

### Decision 3: 保留冷启动元数据查询

GoFrame 首次访问表结构会读取 PostgreSQL 元数据，日志中耗时较明显，但它不是每次首页加载稳定重复的主要来源。本轮只记录为后续可选优化，不通过预热机制增加启动流程复杂度。

## Risks / Trade-offs

- 会话校验改为读取记录后再判断，必须确保租户不匹配仍拒绝、过期会话仍删除、近期活跃不会写入。
- 请求级或列表级 release 快照只能减少单次请求内重复读取，不能消除所有跨请求重复；这是为了避免缺少分布式失效时的陈旧缓存风险。
- 插件 release 读取路径较多，需通过针对 catalog 的单元测试确认按 ID 与按版本读取能复用同一快照。

## Required Assessments

- i18n：不新增、修改或删除用户可见文案，不需要维护前端运行时语言包、插件 manifest/i18n 或 apidoc i18n。
- 缓存一致性：本轮不新增长 TTL 进程级插件状态缓存；请求级快照随请求结束释放，不跨实例共享，不存在分布式陈旧窗口。会话校验仍以数据库记录为权威数据源。
- 数据权限：不新增或修改业务数据接口；只优化鉴权会话和插件元数据读取内部实现。
- 开发工具跨平台：不新增或修改开发工具、脚本或 CI 入口。
